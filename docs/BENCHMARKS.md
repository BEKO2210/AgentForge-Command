# Performance Benchmarks

> Generated by `node tests/perf-bench.mjs` — real numbers on this machine
> (Node v22.22.2, linux/x64). Re-run to refresh.

## Auto-enter watchdog matcher

The auto-enter watchdog scans a rolling ~1.2 KB tail of each PTY's output for a
permission prompt (`AUTO_PROMPT_RE` in `gui/server.js`). This benchmark proves
the README's **"sub-millisecond hot loop"** claim with a measured figure.

| Metric | Value |
|--------|-------|
| Iterations | 2,000,000 |
| Throughput | 237,761 checks/sec |
| **Per-check latency** | **4.206 µs** (0.00421 ms) |
| Sub-millisecond per check? | ✅ yes |

**Conclusion.** A single matcher check costs about
**4.21 µs** — comfortably sub-millisecond. Real PTYs emit on the order
of 1–10 KB/sec, so the matcher is never the bottleneck regardless of the
optional Rust accelerator.

## The Rust accelerator (`forge-pulse`) — optional, advisory

`forge-pulse` checks the **same** prompt patterns in Rust and is auto-detected
when present. It is an optimisation, **not** a correctness dependency: the JS
matcher above is authoritative and the system runs identically without the
binary (5 Rust unit tests cover the sidecar's parsing/matching). The honest
takeaway: the JS path already satisfies the sub-millisecond budget, so Rust is
a "nice to have" for very high-throughput sessions, not a requirement.

_Method: 2,000,000 iterations over a 1000-entry corpus of realistic ~1.2 KB
tails (25% containing a prompt), after a warmup, timed with
`performance.now()`. Numbers vary with CPU; re-run `npm run test:perf`._
